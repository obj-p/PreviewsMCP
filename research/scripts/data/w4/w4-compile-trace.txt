TRACK A — live incremental-build experiment  (xcodebuild path)
================================================================================
Target: /tmp/w4-compile  — single macOS framework "W4", 61 .swift files:

  trivial structs, no cross-references). Built with isolated derivedDataPath.

Method: snapshot mtimes of all */*.o, apply one edit, incremental rebuild,
diff mtimes to list objects actually recompiled. (No dtrace; SIP on.)

M2 — the incremental swiftc driver invocation (key flags, verbatim subset):
  /Applications/Xcode-26.2.0.app/.../usr/bin/swiftc
    -module-name W4
    @.../W4.SwiftFileList            <- filelist = ALL 61 files
    -c
    -enable-batch-mode
    -incremental
    -output-file-map ...
    -emit-module-path .../W4.swiftmodule
    -experimental-emit-module-separately

  W4.SwiftFileList line count: 61   (full module membership passed every build)

Reading: the driver is handed the full 61-file filelist for type-checking, but
`-incremental` + the output-file-map make swift-frontend rebuild ONLY the
changed file's object. Quoted-command-line confirmation of the handoff's
"all files listed, -primary-file <one>" caveat: path count != recompile scope.
Cross-module case (MultiModuleTest log): each module emits its own
`.swiftmodule` (`-experimental-emit-module-separately`) and downstream modules
consume it via `-I` + explicit `.swiftmodule` input -> prebuilt module reuse.

================================================================================
TRACK B — Xcode Previews thunk artifacts  (canvas hot-reload path)
================================================================================
Source: prior real preview sessions on this host (not driven by W4 experiment):
  DerivedData/PreviewTestApp-*/.../Objects-normal/arm64/
  DerivedData/MultiModuleTest-*/.../Objects-normal/arm64/

Per edited file, the preview pipeline emits (PreviewTestApp, app target):
  ContentView.1.preview-thunk.swift          <- generated thunk SOURCE (1 file)
  ContentView.1.preview-thunk.o              <- its object
  ContentView.1.preview-thunk-launch.o
  ContentView.1.preview-thunk.dia
  vfsoverlay-ContentView.1.preview-thunk.swift.json
  vfsoverlay-ContentView.__XCPREVIEW_THUNKSUFFIX__.preview-thunk.swift.json

VFS overlay (verbatim) — maps the REAL source path to the generated thunk, so
the compiler sees exactly one substituted file:
  {"roots":[{"contents":[{"external-contents":".../ContentView.1.preview-thunk.swift",
    "name":"ContentView.swift","type":"file"}],
    "name":".../PreviewTestApp","type":"directory"}], ...}

Generated thunk source (head) — literals rewritten to design-time calls:
  import func SwiftUI.__designTimeString
  import func SwiftUI.__designTimeInteger          (also Float, Boolean)
  #sourceLocation(file: ".../ContentView.swift", line: 1)
  ...
    VStack(spacing: __designTimeInteger("#7210_0", fallback: 20)) { ... }
    Button(__designTimeString("#7210_1", fallback: "Increment")) { ... }

Generation counter is per-file and bumps per edit:
  ViewLibrary: Previews.0.preview-thunk.swift  AND  Previews.1.preview-thunk.swift

Multi-module scope (MultiModuleTest) — thunk is per-module, per-file:
  FeatureModule.build/.../FeatureView.1.preview-thunk.swift
  ViewLibrary.build/.../Previews.1.preview-thunk.swift

Note: these thunk compiles do NOT appear in DerivedData/*/Logs/Build
.xcactivitylog (grep for `preview-thunk*.swift` in decoded logs = empty). The
preview build service compiles them out-of-band, consistent with the handoff's
XCBBuildService warning. The on-disk artifacts are the primary evidence.

Mechanism (object inspection of ContentView.1.preview-thunk.o):
  dynamic-replacement symbols : 0
  __swift5_replace sections   : 0
  PreviewRegistry.makePreview : 4   <- the thunk's entry point
So the body-edit path is NOT in-place dynamic replacement. The thunk compiles a
fresh `DeveloperToolsSupport.PreviewRegistry` conformance with a new
`makePreview()`; the (respawned) preview host calls it. This matches W3's
respawn-only, no-write_mem dispatch finding — recompile a new entry + respawn,
not patch the old body.

================================================================================
OPEN ITEM — the thunk-compile swift-frontend argv
================================================================================
The exact argv of Apple's thunk compile is not logged and no response file
persists it (XCBBuildService runs it in memory). Status of the three angles:

a) DISK (no canvas needed): single-file SCOPE is already proven by the thunk
   source + VFS overlay (one substituted file) and the PreviewRegistry object.
   No depfile is emitted, so disk does not yield the literal flags.

b) RECONSTRUCTION (autonomous, this host): a faithful single-file thunk compile
   assembled by hand on the W4 project, compiles clean (exit 0) and links the
   design-time externals. This is OUR argv, not Apple's, but shows the shape:
     swift-frontend -c \
       -primary-file Sources/ToDo/ToDoView.swift \
       Sources/ToDo/Filler*.swift            <- rest of module as secondary inputs
       -vfsoverlay vfsoverlay-ToDoView.preview-thunk.json \   <- substitutes 1 file
       -module-name W4 -sdk <macosx> -target arm64-apple-macos14.0 \
       -I <W4.build/.../arm64>               <- prebuilt module dir
       -o thunk-recon.o
   `nm -u` on the result shows the design-time literal functions as externals:
     SwiftUI.__designTimeString<A>(_: String, fallback: A) -> A
     SwiftUI.__designTimeInteger<A>(_: String, fallback: A) -> A

c) LIVE CAPTURE (one manual step): `capture-thunk-compile.sh` (sudo-free ps
   poller, anchored on the swift-frontend/swiftc executable) is verified to
   catch a live frontend whose argv carries a preview thunk:
     /Appl.../swift-frontend -primary-file .../ToDoView.swift \
       -vfsoverlay .../vfsoverlay-...preview-thunk.json -module-name W4 -I ...
   To capture APPLE's exact argv: open a #Preview canvas in Xcode, run the
   script, change one body literal. The script writes the argv to
   w4-thunk-argv.txt. That is the only step needing a GUI canvas.
